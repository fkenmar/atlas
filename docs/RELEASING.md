# Releasing & release notes

Releases are automated by [cargo-dist](https://opensource.axo.dev/cargo-dist/)
via `.github/workflows/release.yml` (autogenerated — **don't hand-edit it**;
regenerate with `dist generate` if config changes). This doc covers the human
parts: cutting a tag and shipping a *polished* release note.

## How a release happens

1. Bump the version in `Cargo.toml` (and `pyproject.toml` if it pins one).
2. Add the release's section to `CHANGELOG.md` (see below — this becomes the
   release body).
3. Commit, then push a version tag: `git tag v0.3.0 && git push origin v0.3.0`.
4. The Release workflow builds archives + installers for every target, generates
   checksums, creates the GitHub Release, and attaches the artifacts.

The release **notes are auto-generated by dist from the matching `CHANGELOG.md`
section** (`announcement_github_body`). So the quality of what people read is the
quality of your changelog entry — there's no separate notes step to forget.

## Make the auto-generated note actually sell the tool

The first impression on the releases page matters (see
[`release-notes-draft.md`](release-notes-draft.md) for a full value-first
template). Practical rule: **the top of each `CHANGELOG.md` release section should
lead with the value, not a raw commit list.** A good shape:

```markdown
## [0.3.0] - 2026-07-01

Give your coding agent a map of your repo so it stops burning tokens finding its
way around. Same answers, ~65% fewer tokens (reproducible — see benchmark/README).

### Added
- ...
### Changed
- ...
### Fixed
- ...
```

dist will lift that whole section into the release body, so the lead paragraph
travels with it. Keep benchmark wording in sync with
[`benchmark/history.md`](../benchmark/history.md) and **never** add an edit-task
token claim without a result that clears the variance gate in
[`benchmark/README.md`](../benchmark/README.md) (the launch checklist enforces
this).

## Pre-release checklist

The full gate lives in [`LAUNCH_CHECKLIST.md`](LAUNCH_CHECKLIST.md). The
essentials:

- `cargo fmt --all && cargo clippy --all-targets -- -D warnings && cargo test` green.
- README install commands and badges match the new tag.
- `CHANGELOG.md` has the section, value-first, with honest benchmark wording.
- Alpha/stability note present for pre-1.0 tags (dist marks `-alpha`/`-prerelease`
  tags as prereleases automatically).

## After the release

- Announce from [`social-launch.md`](social-launch.md) and work the
  [`submission-drafts.md`](submission-drafts.md) packets.
- If the GitHub Pages site is enabled, it serves the latest `docs/` automatically
  — no release step needed.
